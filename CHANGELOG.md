# Changelog

Формат основан на [Keep a Changelog](https://keepachangelog.com/),
проект придерживается [Semantic Versioning](https://semver.org/).

## [0.10.0] - 2026-05-16

### Added

- Команда `/roast` (и `/roast @user`) — бот в активной персоне чата делает
  личный стендап-роаст на основе последних 30 сообщений цели и её профиля
  из `UserMemoryProfile`. Кулдаун 1/день на чат
- Промт-режим «острый, без сюсюканья, маты по делу, ничего не табу кроме
  явной агрессии и того, что цель отметила в `boundaries` как скрытое».
  Активная персона чата подмешивается сверху и усиливает/смягчает baseline
- Защита: `/roast @bot` — отказ; `/roast` самого себя — отказ; цель без
  сообщений за неделю — отказ; в личке — отказ

### Internal

- Новая таблица `roast_runs` + миграция `20260516_03_roast_runs`
- Сервис `RoastService`: per-chat `asyncio.Lock`, two-provider LLM-fallback
  через `_llm_call`, plain-text fallback при сбое всех провайдеров (кулдаун
  тогда НЕ списывается), `boundaries` подмешиваются отдельным блоком
  `Hidden topics` в user-промпт
- Тесты: 33 новых кейса (унификация cooldown 7 дней, race serialization
  через lock, проверка ROAST_RULES в системном промпте, выбор random
  с исключением initiator/бота, и т.д.)
- Single-process assumption задокументирован в коде (mirror того, что
  сделано в `MonthlyChampionService`)

## [0.9.0] - 2026-05-16

### Added

- Игра «🎲 Кости»: команда `/dice` (или кнопка в меню `/games`) — игрок
  выбирает 1 или 2 числа от 1 до 6 и бросает анимированный кубик Telegram.
  При совпадении списываются очки в месячной рулетке: −2 за одно число
  (шанс ≈17%), −1 за два (шанс ≈33%). Один бросок в день на игрока в чате.

### Internal

- Новая таблица `dice_rounds` + миграция `20260516_01_dice_game`
- Сервис `DiceGameService` с атомарной записью раунда и `RouletteScoreAdjustment`
  под блокировкой по `(chat_id, user_id)` и rechec'ом дневного лимита внутри
  транзакции
- Расширение `app/bot/router_games.py`: парсер callback'ов, билдер inline-клавиатуры,
  диспатч-хэндлер для toggle/cancel/roll с защитой от чужих нажатий
- Тесты: 22 новых кейса (юнит payout-таблицы, сервисные интеграции, парсер
  callback'ов, smoke-тесты `_open_dice` и end-to-end диспатча с моками TG bot)

## [0.8.0] - 2026-05-10

### Added

- Ежемесячный итог рулетки: 1-го числа в 12:00 MSK бот в стиле активной
  персоны оглашает «Короля X месяца» по результатам рулетки за прошлый
  месяц. Звание выводится из дневного титула чата (например, «Мудак дня»
  → «Король Мудаков месяца»). Победитель закрепляется в фактах чата
  и подмешивается в системный промпт до следующего оглашения, поэтому LLM
  обыгрывает титул в обычной переписке
- При ничье на вершине проводится драматичный runoff в три сообщения
  (анонс ничьи → «бросаем кости» → оглашение через `random.choice`)
- Если за месяц в чате не было ни одного `/roll`, бот иронично
  комментирует пустой месяц (только если бот включён администратором)
- Если победитель уже покинул чат — объявление заочное по последнему
  снэпшоту username
- Catch-up при старте бота: если cron не отработал (downtime),
  оглашение происходит при первом запуске в пределах первой недели месяца
- Админ-команда `/roulette_force_summary` для ручного запуска подведения
  итогов в текущем чате (только для админов группы)

### Internal

- Новая таблица `monthly_champions` (миграция `20260510_01`),
  колонка `chat_memories.monthly_champion` — отдельный JSON-слот, не
  конкурирует с `lore`/`members`
- `RouletteService._aggregate` расширен параметром `end` для замкнутого
  диапазона, чтобы майские wins не попадали в апрельский итог
- Сервис `MonthlyChampionService`: per-chat asyncio.Lock, идемпотентность
  через `UNIQUE(chat_id, period_start)`, silent text-fallback при падении
  обоих LLM-провайдеров, корректный дрейф между Москвой и UTC
- Покрытие: 32 unit/integration теста + E2E через реальный
  `RouletteService._aggregate`. `make check` (ruff + mypy + 241 тест) — зелёный

## [0.6.4] - 2026-04-23

### Fixed

- Рулетка больше не приписывает победителю чужие реплики. Ранее LLM не знал, кто именно выиграл, и мог обыгрывать звание через сообщения других участников, создавая путаницу в авторстве. ## [0.4.1] - 2026-04-14

### Fixed

- LLM fallback теперь срабатывает и на network errors (connection refused, DNS, timeout, reset), не только на 429/5xx. В v0.4.0 наблюдали как OpenRouter был временно недоступен на TCP-уровне — fallback на OpenAI не активировался, пользователь не получал ответ. Исправлено: если primary-провайдер вообще не отвечает, бот переключается на sibling.

## [0.4.0] - 2026-04-14

### Added

- Бот теперь может отвечать голосовыми сообщениями. По умолчанию редко (~5%), но если пользователь reply-нул боту голосовым — шанс выше (~60%, разговорный обмен голосами). Используется OpenAI TTS (`gpt-4o-mini-tts`) с opus-форматом, родным для Telegram. Применяется к direct-address, спонтанным интерджектам и revive.
- Маппинг персона → голос: каждая из 6 персон (gopnik, chatmate, standup, boss, zoomer, jarvis) озвучивается своим характерным OpenAI voice. Дефолты подобраны под персону, переопределяются в админке.
- Админ-раздел «Голосовые ответы»: kill-switch (`tts_enabled`), вероятности (`tts_reply_p` / `tts_voice_reply_p`), дневной лимит (`tts_daily_limit`), per-persona voice dropdown'ы.

### Internal

- Новые сервисы: `app/services/llm/tts.py` (OpenAI `/v1/audio/speech`), `app/bot/voice_reply.py` (`send_reply_maybe_voice` + `send_chat_maybe_voice` для voice-or-text).
- `SpontaneityPolicy.should_reply_with_voice` — decision-метод с буст-вероятностью на reply-to-bot голосовые.
- 9 новых дефолтов в `app_config` для TTS-настроек.
- При TTS-ошибке (5xx, 4xx, network) — silent fallback на текст. Voice excuses (когда Whisper не сработал) остаются текстом по дизайну.

## [0.3.0] - 2026-04-14

### Added

- Бот понимает голосовые сообщения и круглые видео. Задай вопрос голосовой с @упоминанием или в ответ к боту — он транскрибирует через OpenAI Whisper и ответит как на обычный текст. Плюс бот может внезапно прокомментировать чужое голосовое в чате, если сработает кубик интерджекта.
- Reply-chain: бот понимает голосовые из треда. Если пользователь отвечает на чужое голосовое и обращается к боту, тот транскрибирует исходное сообщение. Транскрипт кешируется в БД — повторно за Whisper не платит.
- Админ-параметры голосовых: глобальный kill-switch `voice_enabled`, максимальная длина (`voice_max_seconds`, 0 = без лимита), дневной лимит распознаваний на чат (`whisper_daily_limit`, 0 = без лимита). Под каждым полем — справка.

### Changed

- При ошибках распознавания (Whisper недоступен, голосовое слишком длинное, исчерпан лимит) бот не отвечает канифольным «ошибка» — генерирует отговорку через LLM в текущем стиле персоны.

### Internal

- Новые сервисы `app/services/llm/whisper.py` (Whisper API) и `app/services/reply_voice.py` (reply-chain + кеш). Handler `_handle_voice_message` с engage-then-transcribe логикой. Хранение `tg_file_id` для voice/video_note без новых миграций. Правки mypy override для транзитивных импортов из тестов.

## [0.2.1] - 2026-04-14

### Internal
