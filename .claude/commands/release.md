# /release — подготовить новый релиз

Выполни релиз проекта Gremlin Bot: проанализируй коммиты, подними версию, обнови changelog и, если есть user-facing изменения, подготовь текст рассылки. Рассылка в Telegram происходит автоматически при старте бота после деплоя — твоя задача только подготовить `RELEASE_NOTES.md` и запушить.

## Контекст

- Версия хранится в `pyproject.toml` → `[project].version`
- Технический журнал: `CHANGELOG.md` (формат Keep a Changelog)
- Текст рассылки: `RELEASE_NOTES.md` — если не пустой, бот при старте разошлёт его во все активные чаты. Если пустой — рассылки нет, но теги и changelog всё равно создаются
- Сообщения шлются plain text, без Markdown и HTML. Telegram HTML-форматирование не применяется
- Проект уже использует GitHub Actions CD: push в `main` → автодеплой по SSH → `docker compose up -d --build`. Ты делаешь только push, остальное произойдёт само

## Последовательность действий

### 1. Убедись в чистом состоянии

```bash
git status
git rev-parse --abbrev-ref HEAD
```

- Должен быть `main`, без uncommitted/untracked файлов, которые связаны с релизом
- Если user на другой ветке — спроси, действительно ли релизим отсюда
- `git pull --ff-only` чтобы быть в актуальном состоянии

### 2. Определи стартовую точку

```bash
git describe --tags --abbrev=0 2>/dev/null || echo "NO_TAGS"
```

- Если есть тег (`v0.1.0` и т.п.) — работай от него
- Если нет — от начала истории
- Прочитай текущую версию из `pyproject.toml` (ключ `[project].version`)

### 3. Собери коммиты для анализа

```bash
git log <prev_tag>..HEAD --pretty=format:"%H%x00%s%x00%b%x1e" --no-merges
```

Для каждого коммита посмотри ещё и diff, чтобы классифицировать не на основании commit message (они могут быть неточными), а по реальным изменениям:

```bash
git show --stat <hash>
```

Если diff маленький — `git show <hash>`. Если большой — читай через Read только изменённые файлы.

### 4. Классифицируй каждый коммит

Присвой каждому коммиту одну категорию:

| Категория | User-facing | Что относится |
|-----------|-------------|---------------|
| `feat` | ✅ | новая команда Telegram, новая кнопка в админке, новая настройка чата, новый игровой функционал |
| `fix` | ✅ | исправление бага, который видел пользователь (ошибки в ответах, упавшие команды, зависшая рулетка) |
| `improvement` | ✅ | изменение поведения существующей команды, улучшение качества ответов, UX-правки в админке |
| `internal` | ❌ | рефакторинг, типизация, lint/mypy, тесты, CI, миграции без видимого эффекта, docs, README, чистка мёртвого кода, правки зависимостей |

**Ключевой вопрос при классификации**: «Заметит ли это пользователь чата или админ веб-панели?». Если нет — `internal`.

Граничные случаи:
- Правка промпта персоны — `improvement` (пользователь заметит тон)
- Правка лимитов, таймаутов — `internal`, если не затрагивает UX; `improvement`, если заметно
- Изменения в миграции — `internal`
- Новое поле в админ-форме — `feat`
- Удаление устаревшей команды — `feat` (breaking) или `improvement`

### 5. Определи следующую версию (semver)

- Если есть хотя бы один коммит с `BREAKING CHANGE:` в теле или префиксом `feat!:`/`fix!:` → **major** (0.2.0 → 1.0.0)
- Иначе если есть хотя бы один `feat` → **minor** (0.1.0 → 0.2.0)
- Иначе если есть `fix`/`improvement` → **patch** (0.1.0 → 0.1.1)
- Если все коммиты `internal` → **patch**

### 6. Обнови `pyproject.toml`

Подними `version` в секции `[project]` на рассчитанное значение.

### 7. Обнови `CHANGELOG.md`

Добавь новую секцию **над** предыдущими версиями. Формат:

```markdown
## [X.Y.Z] - YYYY-MM-DD

### Added
- описание нового функционала

### Changed
- описание изменений

### Fixed
- описание исправлений

### Internal
- краткий список технических правок
```

- Пропускай пустые подсекции
- Пиши короткими императивными фразами: «Добавить команду `/quiz`», «Исправить зависание рулетки при пустом чате»
- `Internal` можно собрать в один пункт: «Рефакторинг сервиса рассылки, правка типизации, обновление CI»

### 8. Подготовь `RELEASE_NOTES.md`

**Если есть хотя бы один user-facing коммит (feat / fix / improvement):**

Напиши короткий дружелюбный текст (3-10 строк), plain text, без Markdown и без HTML. Формат:

```
🎉 Обновление vX.Y.Z

Новое:
• короткое описание фичи

Улучшения:
• что поменялось

Исправлено:
• какой баг починили
```

Правила:
- Не упоминай хеши, internal-правки, названия файлов, названия функций
- Пиши для обычного пользователя чата, а не для разработчика
- Если в релизе есть breaking change — сразу об этом, первой строкой после заголовка
- Эмодзи можно, но умеренно (1-3 на всё сообщение)
- Никаких упоминаний Claude, AI, автогенерации

**Если все коммиты internal:**

Сделай файл пустым: `echo -n > RELEASE_NOTES.md`. Бот при старте увидит пустоту и не будет рассылать.

### 9. Прогони проверки качества

**Обязательный шаг перед любым показом пользователю и коммитом.** Запусти полный локальный набор проверок:

```bash
make check
```

Это эквивалентно `ruff check . && mypy app tests && pytest`. Если `make` недоступен, запускай по отдельности через venv (`.venv/bin/ruff check .`, `.venv/bin/mypy app tests`, `.venv/bin/pytest`).

- **Если любая проверка упала** — не показывай релиз пользователю и не коммить. Сначала почини:
  - `ruff` → применяй `ruff check --fix .` если ошибки автоматически исправимы, иначе правь руками
  - `mypy` → исправляй типы, добавляй аннотации; не глуши `# type: ignore` без необходимости
  - `pytest` → смотри упавшие тесты, чини код или тест (если тест реально неверный)
- После исправления **перезапусти `make check` полностью**. Релиз не уходит пока все три проверки не будут зелёными
- Если ошибка в коде, который внесла не текущая сессия, а был в `main` до неё — сообщи об этом пользователю и спроси, чинить ли здесь

### 10. Покажи результат пользователю

Выведи в чат:
- Предыдущая и новая версия
- Список коммитов с классификацией
- Добавленный блок changelog
- Подготовленный `RELEASE_NOTES.md` (или явное сообщение, что рассылки не будет)

Жди подтверждения. **Не коммить сам до одобрения.** Если пользователь попросит правки — применяй.

### 10. После подтверждения — коммит и пуш

```bash
git add pyproject.toml CHANGELOG.md RELEASE_NOTES.md
git commit -m "chore(release): vX.Y.Z"
git tag vX.Y.Z
git push --follow-tags
```

- Автор коммита — настройки git пользователя (не добавляй Co-Authored-By для Claude)
- Если в процессе были другие коммиты в этом же запуске — используй их авторство и conventional-формат (`feat:`, `fix:`, `chore:`, `docs:`, `refactor:`, `test:`, `ci:`), но **без** Co-Authored-By

### 11. Сообщи о завершении

Выведи:
- Новый тег
- Ссылку на GitHub Actions workflow run (если знаешь URL репозитория)
- Будет ли рассылка (да/нет) и во сколько чатов приблизительно (не критично)

## Не делай

- Не создавай релиз, если `git status` показывает неотслеживаемые файлы с подозрительными именами (`.env.prod`, `dump.sql`) — спроси
- Не делай `git push --force`
- Не указывай себя соавтором в коммитах
- Не добавляй в `RELEASE_NOTES.md` ничего, что не увидит пользователь: числа покрытия, coverage, имена файлов, git-хеши
- Не трогай версию в `app/main.py` — она теперь читается из `pyproject.toml` через `get_version()`
