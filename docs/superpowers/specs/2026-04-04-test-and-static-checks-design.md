# Test And Static Checks Design

## Goal

Добавить в проект базовый, но практически полезный контур качества: автоматические тесты на ключевую бизнес-логику, статический анализ через `ruff` и `mypy`, а также понятную документацию по локальному прогону всех проверок.

## Scope

В рамках этой итерации покрываются:

- `pytest` для сервисов с детерминированной логикой и минимальной зависимостью от Telegram/API;
- `ruff` для линтинга, проверки импортов и базового форматирования;
- `mypy` в pragmatic-конфигурации для `app/` и `tests/`;
- команды в `Makefile` и раздел в `README.md` для запуска проверок;
- CI-workflow для автоматического запуска quality checks на GitHub Actions.

В рамках этой итерации не покрываются:

- полноценные end-to-end сценарии Telegram webhook/polling;
- глубокие интеграционные тесты aiogram-роутеров;
- полная типизация всех внешних библиотек без точечных `ignore`/override.

## Success Criteria

- В репозитории появляется исполняемый набор тестов на `pytest`.
- Тесты покрывают минимум следующие области:
  - сборку prompt/history в `app/services/context.py`;
  - кэширование и чтение настроек в `app/services/settings.py`;
  - чтение/кэширование app config в `app/services/app_config.py`;
  - дневные лимиты и refund в `app/services/usage_limits.py`;
  - детерминированные части `app/services/roulette.py`, не требующие реального Telegram API.
- В репозитории появляется единая конфигурация для `ruff`, `mypy` и `pytest`.
- Локально можно выполнить отдельные команды `lint`, `typecheck`, `test` и агрегированную `check`.
- В GitHub Actions появляется workflow, который автоматически запускает эти проверки.
- В `README.md` описано, как устанавливать dev-зависимости и как запускать проверки.

## Architectural Approach

### 1. Testing strategy

Основной упор делается на сервисный слой, потому что именно там сосредоточена наиболее ценная и сравнительно стабильная логика.

Тесты делятся на два типа:

- чистые unit-тесты без I/O, где можно проверять поведение напрямую;
- лёгкие async/integration-like тесты сервисов, которым нужен SQLAlchemy sessionmaker или Redis-подобный интерфейс.

Чтобы тесты были быстрыми и независимыми:

- для Redis используются простые in-memory test doubles;
- для SQLAlchemy используется in-memory SQLite с созданием схемы на лету;
- Telegram bot и LLM-зависимости мокируются точечно только там, где это необходимо.

### 2. Initial test targets

Первая волна покрытия включает:

- `build_messages()` и `build_system_prompt()`:
  - фильтрация service messages;
  - пропуск команд вида `/...`;
  - объединение последовательных сообщений одного автора;
  - приоритет `closing_text` и текущего пользовательского сообщения;
  - trimming по `max_tokens`;
  - корректная подстановка `focus_text`.
- `SettingsService`:
  - возврат значений по умолчанию;
  - чтение значения из Redis cache;
  - чтение из БД при промахе кэша;
  - `set()` создаёт чат и инвалидирует cache.
- `AppConfigService`:
  - возврат merged defaults;
  - чтение из cache;
  - обновление значения и сброс cache.
- `UsageLimiter`:
  - успешный `consume()`;
  - отклонение при превышении лимита;
  - корректный `refund()`;
  - расчёт ключей по текущему дню и timezone-aware логике.
- `RouletteService`:
  - фильтрация bot-usernames при регистрации;
  - обновление username существующего участника;
  - `participant_count()` не считает bot-аккаунты;
  - выбор custom title без обращения к LLM;
  - простые негативные сценарии `roll()` без доставки сообщения.

### 3. Static analysis strategy

`ruff` используется как основной быстрый quality gate:

- lint по стилю, импортам и очевидным ошибкам;
- возможность автофикса локально;
- минимально спорная конфигурация, чтобы не застрять на косметике.

`mypy` вводится постепенно:

- первая итерация проверяет `app/` и `tests/`;
- для библиотек без качественных stubs допускаются точечные настройки `ignore_missing_imports`;
- если в проекте встретятся небольшие типовые дефекты, они исправляются в рамках внедрения;
- если типизация отдельного модуля окажется слишком дорогой, допускается локальное смягчение, но не полное отключение всей проверки.

### 4. Configuration layout

Чтобы не распылять настройки по нескольким файлам, конфигурация по возможности централизуется в `pyproject.toml`.

В `pyproject.toml` планируется разместить:

- настройки `pytest`;
- настройки `ruff`;
- настройки `mypy`;
- при необходимости общие project metadata для dev-инструментов.

Если в ходе внедрения окажется, что отдельному инструменту удобнее иметь собственный файл-конфиг, это допускается только при явной практической причине.

### 5. Developer workflow

Для локальной разработки вводятся явные команды:

- `make lint`
- `make typecheck`
- `make test`
- `make check`

Дополнительно допускается команда для автофиксов линтера, если она не усложнит Makefile.

README должен описывать:

- какие dev-зависимости нужны;
- как их установить;
- как запускать отдельные проверки;
- как интерпретировать агрегированную команду.

### 6. CI strategy

Помимо существующего deploy workflow добавляется отдельный workflow качества, который запускается:

- на `push`;
- на `pull_request`.

Он должен:

- ставить Python подходящей версии;
- устанавливать runtime и dev-зависимости;
- последовательно запускать `ruff`, `mypy` и `pytest`.

Deploy workflow при этом не должен смешиваться с quality-проверками: quality pipeline живёт отдельно и служит gate-ом для изменений.

## Files Expected To Change

- новый `pyproject.toml` с настройками инструментов;
- `requirements.txt` или отдельный dev-requirements файл для test/lint/typecheck зависимостей;
- `Makefile` с командами quality checks;
- `README.md` с разделом про проверки;
- новый workflow в `.github/workflows/` для CI-проверок;
- директория `tests/` с fixtures, test doubles и целевыми наборами тестов;
- точечные правки в `app/`, если они понадобятся для прохождения `mypy` или повышения тестопригодности.

## Risks And Mitigations

- `mypy` может выявить много исторических проблем.
  - Смягчение: pragmatic config, точечные overrides, исправление только наиболее близких к внедряемым модулям проблем.
- Некоторые сервисы завязаны на async SQLAlchemy и Redis.
  - Смягчение: выделить простые test doubles и reuse-фикстуры, чтобы тесты не стали тяжёлыми.
- Импорт `app.main` может тянуть реальные внешние зависимости на старте.
  - Смягчение: на первой итерации не делать тяжёлые app-wide smoke tests, если они требуют большого числа инфраструктурных моков.

## Test Plan

После реализации должны выполняться:

- `make lint`
- `make typecheck`
- `make test`
- `make check`

В CI запускается тот же набор проверок без специальных ручных шагов.

## Out Of Scope Follow-Ups

Следующей итерацией можно добавить:

- smoke tests FastAPI health/metrics endpoints;
- тесты роутеров aiogram;
- coverage reporting в CI;
- pre-commit hooks для локального quality gate.
