# Пилот персональной памяти

## Что появилось
- Память по ключу `(chat_id, user_id)`, а не только окно последних сообщений.
- Три слоя памяти:
  - `messages` — сырая история, по которой работает user-specific retrieval;
  - `user_memory_profiles` — компактный профиль пользователя в рамках чата;
  - `relationship_states` — накопленное состояние отношений бота с участником.
- Retrieval памяти перед ответом бота:
  - при упоминании бота;
  - при reply на сообщение бота;
  - при спонтанном вмешательстве;
  - при оживлении тихого чата.

## Что нужно для пилота
- Тестовый бот Telegram.
- Рабочий `OPENAI_API_KEY` или другой OpenAI-compatible endpoint для основной модели.

## Переменные окружения
- `OPENAI_API_KEY`
- `OPENAI_API_BASE`

Глобальные runtime-настройки включаются в админке:
- `user_memory_enabled`
- `memory_sidecar_enabled`
- `memory_top_k`
- `memory_max_prompt_tokens`
- `memory_rag_candidate_limit`
- `memory_group_users`

## Как раскатить поверх старой версии
1. Обновить код и зависимости.
2. Запустить `alembic upgrade head`.
3. Перезапустить бота.

Старая таблица `messages` и остальные сущности не меняются. Новая версия встаёт поверх старой схемы и начинает заполнять новые таблицы постепенно.

## Рекомендуемый сценарий пилота
1. Поднять отдельный тестовый бот и отдельный тестовый чат.
2. В админке включить `user_memory_enabled` и `memory_sidecar_enabled`.
3. В настройках тестового чата оставить `personalization_enabled=true`.
4. Накормить бота явными сообщениями от 2-3 пользователей:
   - `меня зовут Антон`
   - `я люблю короткие ответы`
   - `не называй меня по имени`
   - `я работаю над телеграм-ботом`
5. Открыть в админке:
   - `/admin/chats/<chat_id>/memory`
   - `/admin/chats/<chat_id>/memory/<user_id>`
6. Проверить, что у пользователя появились:
   - profile summary;
   - relationship state.
   - использование его старых сообщений в retrieval.

## Что должно работать
- После `@bot` бот должен учитывать профиль автора сообщения.
- После `@bot` бот должен уметь подтягивать релевантные старые сообщения этого пользователя из `messages`.
- После reply боту он должен помнить предыдущие предпочтения этого же автора.
- При случайном вмешательстве бот должен учитывать target-пользователя последней реплики.
- При revive тихого чата бот должен подтягивать краткую память по 1-3 недавним участникам.
- В админке должен работать просмотр и reset памяти конкретного пользователя.

## Как тестировать руками
### 1. Проверка identity / preference / boundary
- Пользователь A пишет:
  - `меня зовут Антон`
  - `я люблю очень короткие ответы`
  - `не называй меня Антошей`
- Потом пишет `@bot как думаешь?`
- Ожидаемо: бот чаще отвечает коротко и не использует запрещённое обращение.

### 2. Проверка project memory
- Пользователь B пишет:
  - `я работаю над tg-ботом на fastapi`
- Через несколько сообщений обращается:
  - `@bot что мне дальше сделать с этим проектом?`
- Ожидаемо: бот связывает вопрос с ранее упомянутым проектом.

### 3. Проверка relationship state
- Пользователь несколько раз дружелюбно общается с ботом.
- Потом несколько раз грубо.
- В админке должны меняться `affinity`, `familiarity`, `tension`.

### 4. Проверка fallback
- Выключить `user_memory_enabled` в глобальной админке.
- Поведение должно откатиться к старой логике: только недавний контекст без персональной памяти.

## На что смотреть в пилоте
- Не слишком ли много нерелевантных сырых сообщений попадает в retrieval.
- Не слишком ли сильно раздувается prompt.
- Не становится ли бот слишком самоуверенным из-за старых фактов.
- Достаточно ли retrieval вытаскивает нужные прошлые сообщения при реальных вопросах.

## Что я бы оценивал после пилота
- hit-rate полезной памяти на mention/reply;
- число лишних сообщений в retrieved context;
- среднюю задержку ответа после включения retrieval + sidecar;
- насколько часто нужен ручной reset памяти через админку.
